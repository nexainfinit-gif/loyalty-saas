## Summary

**Overall Assessment: Proceed with conditions**

The platform demonstrates moderate GDPR awareness with functional consent tracking, unsubscribe mechanisms, and multi-tenant data isolation. However, several critical gaps were identified around consent documentation, data retention, third-party disclosures, and a column naming inconsistency that undermines consent tracking reliability.

## 2. Critical Findings

### CRITICAL-01: Column naming inconsistency breaks consent tracking

**Risk: High** — Data integrity failure in consent records

Two registration routes use different column names for the same field: `marketing_consent` vs `consent_marketing`. The unsubscribe endpoint has a workaround that tries both names, confirming this is a known but unresolved issue.

**Impact:** Consent state may not be reliably queried, potentially sending marketing emails to non-consented users or failing to record consent properly.

**Recommendation:** Standardize to a single column name via database migration. Audit existing records for data integrity.

### CRITICAL-02: QR tokens exposed to QuickChart (third party)

**Risk: High** — Customer identifier leaked to external service

QR codes in emails are generated by passing customer scan URLs (containing the QR token) to `quickchart.io` as a GET parameter. This exposes the unique customer identifier to a third party via URL, visible in their server logs.

**Impact:** QuickChart could theoretically reconstruct scan URLs. The QR token is effectively a bearer credential for receiving loyalty points.

**Recommendation:** Generate QR code images server-side (e.g., using `qrcode` npm package) to eliminate third-party token exposure.

### CRITICAL-03: No consent text versioning

**Risk: High** — Cannot prove informed consent under GDPR Art. 7

Consent is recorded as a boolean with timestamp and IP, but the actual text the user agreed to is not stored or versioned. If the consent wording changes, there is no way to prove what each user consented to.

**Recommendation:** Store a `consent_text_version` identifier with each consent record. Maintain a version history of consent text.

### CRITICAL-04: Audit logs retain deleted customer PII

**Risk: Medium-High** — Undermines right to erasure

When a customer is deleted, the audit log entry retains their email and name in the `metadata` JSONB field. This means PII persists indefinitely after a deletion request.

**Recommendation:** Anonymize or hash PII in audit log metadata. Store only a pseudonymous reference (e.g., customer UUID) rather than email/name.

## 3. High-Priority Findings

### HIGH-01: No data retention policy

No TTL or automated cleanup exists for scan events, transactions, audit logs, or wallet pass records. Data accumulates indefinitely, violating the GDPR storage limitation principle (Art. 5(1)(e)).

**Recommendation:** Define and enforce retention schedules:
- Transactions: 7 years (Belgian tax/accounting requirements)
- Scan events: 2 years
- Audit logs: 1 year (anonymize PII after)
- Implement automated cron-based cleanup

### HIGH-02: Privacy policy incomplete

The privacy page at `/privacy` exists and covers major processors (Supabase, Resend, Stripe, Vercel, Google/Apple Wallet) but omits:
- **Cloudflare Turnstile** — used for bot detection at registration, sends IP and user-agent to Cloudflare
- **QuickChart** — receives customer QR tokens in URL parameters
- **Apple Push Notification Service** — receives pass update notifications
- **IP tracking** for rate limiting purposes
- **Audit log retention** practices
- **Data retention periods** for each data category

### HIGH-03: No Terms of Service

No Terms of Service file was found in the codebase. For a B2B SaaS, this is essential to define the relationship with restaurant owners, liability, acceptable use, and data ownership.

### HIGH-04: Email verification tokens never expire

Verification tokens (`email_verification_token`) have no TTL. An old token from a leaked or forwarded email could be used indefinitely.

**Recommendation:** Add a 24–48 hour expiry on verification tokens.

### HIGH-05: Data minimization concerns

- **Postal code** is collected but never used in any feature
- **Phone number** (slug registration route) is collected but never used
- Both violate the GDPR data minimization principle (Art. 5(1)(c))

**Recommendation:** Either remove these fields or document their purpose. If kept for future use, obtain explicit consent with a stated purpose.

## 4. Medium-Priority Findings

### MED-01: No GDPR cookie consent banner

No cookie consent mechanism (OneTrust, Cookiebot, etc.) was detected. While the app primarily uses session cookies (Supabase auth), the Cloudflare Turnstile integration may set tracking cookies.

### MED-02: No data portability API

GDPR Art. 20 requires data portability in a structured, machine-readable format. The CSV export (`/api/export-csv`) covers basic customer data but does not include transaction history, scan events, or wallet data — and is only accessible to restaurant owners, not to data subjects themselves.

### MED-03: Wallet pass data lifecycle

When a customer is deleted, wallet passes are marked as "revoked" but the database rows persist. Additionally, passes already on customers' phones (Apple/Google) cannot be remotely deleted — they can only be "voided."

### MED-04: No breach notification procedure

No documented data breach response procedure exists in the codebase. GDPR Art. 33 requires notification to the supervisory authority (Belgian APD) within 72 hours.

### MED-05: No DPIA documented

Given the processing of customer loyalty data across multiple restaurants (systematic monitoring of customer behavior), a Data Protection Impact Assessment is likely required under GDPR Art. 35.

## 6. Strengths

The platform does several things well from a compliance perspective:

- **One-click unsubscribe** is implemented in every marketing email, with a public endpoint that requires no authentication
- **Consent tracking** records timestamp and IP at the moment of consent
- **Multi-tenant isolation** is enforced at database level (RLS policies) and application level (query scoping by `restaurant_id`)
- **Customer deletion** cascades properly: removes customer record, revokes wallet passes, deletes transactions
- **Audit logging** captures security-relevant actions (deletions, campaigns, pass changes)
- **No credit card data** touches the application — Stripe handles all payment processing
- **CAPTCHA protection** at registration prevents automated abuse
- **Campaign consent filtering** checks `consent_marketing` before sending any email
- **CRON endpoint protection** uses timing-safe secret comparison

The following areas should be reviewed by qualified legal counsel:

- **Cookie classification** — determine if Turnstile cookies are "strictly necessary" or require consent
